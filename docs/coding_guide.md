# AI 应用评论编码指南

本文档用于指导 `data/clean_reviews.csv` 的人工标注，目标是把公开应用商店评论转化为可分析的 AI 应用产品体验数据。

## 数据与样本来源

- 主分析对象固定为三款 AI 应用：豆包、DeepSeek、Kimi。
- iOS 侧样本来自 App Store。
- Android 侧样本优先使用应用宝。
- 某个 App 在应用宝上的有效评论较少或低信息量评论较多时，可以使用小米应用商店作为 Android 侧补充来源。
- 不同 Android 应用商店的用户结构、评论数量和评论质量可能存在差异，补充样本用于观察不同来源的反馈特征。
- 采样方式为手动公开采样，数据处理保留评论分析所需字段。

## 编码原则

- 根据评论文本和可见元数据编码，缺失信息保持为空。
- 保留用户原意，不把评论改写成研究者观点。
- 每条评论优先标一个主要问题分类。
- 如果评论包含多个问题，选择对用户体验影响最大的一个，并在 `notes` 中说明次要问题。
- 标签不确定时使用较保守的标签，并在 `notes` 中记录不确定原因。
- 示例行用于说明格式，正式分析时替换为真实公开评论。
- 点餐、奶茶、购物、配送、安装包大小等明显不指向 AI 应用产品体验的评论，归档到 `data/excluded_reviews.csv`。

## usage_scene 取值

| 取值 | 使用说明 |
| --- | --- |
| 写作 | 文案、提纲、改写、总结、邮件、报告等写作任务 |
| 搜索 | 查资料、问事实、找答案、信息整理 |
| 学习 | 作业、备考、知识解释、语言学习 |
| 办公 | 会议纪要、表格、汇报、职场任务 |
| 聊天 | 陪伴、闲聊、情绪表达、泛对话 |
| 图片/视频 | 图片生成、视频生成、视觉相关能力 |
| 代码 | 编程、调试、代码解释、技术问答 |
| 综合/不明确 | 场景不清楚，或同时覆盖多个场景 |

## issue_category 一级问题分类

| 取值 | 定义 | 示例判断 |
| --- | --- | --- |
| 模型能力问题 | 与模型回答、理解、推理、生成质量直接相关的问题 | 回答不准、编造信息、复杂指令理解失败 |
| 交互体验问题 | 与入口、流程、界面、引导、使用路径相关的问题 | 功能入口难找，新用户不知道怎么开始 |
| 性能与稳定性问题 | 与响应速度、卡顿、崩溃、加载失败相关的问题 | 回答慢、卡住、闪退 |
| 会员与商业化问题 | 与会员权益、价格、付费限制、转化体验相关的问题 | 免费次数少，开会员后权益不清楚 |
| 内容安全与合规问题 | 与敏感内容、安全规则、内容审核和合规体验相关的问题 | 敏感内容处理异常，正常内容被误拦截 |
| 账号、隐私与数据问题 | 与账号登录、数据保存、隐私、权限和个人数据有关的问题 | 登录失败，历史记录丢失，担心隐私 |
| 用户预期与产品定位问题 | 用户对产品用途、能力范围和适用场景的理解与预期问题 | 用途理解成本高，预期过高 |

## ai_issue_type AI 特定问题类型

| 取值 | 定义 |
| --- | --- |
| 回答不准 | 回答与事实、题意或用户问题不匹配 |
| 幻觉 / 编造信息 | 生成看似合理但缺少事实依据或验证路径的信息 |
| 指令跟随差 | 未按用户指定格式、限制、语气或任务要求输出 |
| 上下文记忆差 | 多轮对话中忘记前文、重复询问或上下文断裂 |
| 搜索引用不足 | 信息来源、引用、依据不清晰，影响可信度 |
| 生成内容质量不稳定 | 同类任务输出质量波动大，不稳定 |
| 多轮对话体验差 | 追问、纠错、连续任务体验不顺畅 |
| 响应速度慢 | 生成、加载或等待时间过长 |
| 功能入口难找 | 用户不知道某功能在哪里或如何开始 |
| 会员限制影响核心体验 | 会员限制直接影响核心功能使用或价值感知 |
| 不明确/其他 | 评论难以归入上述类型，或信息不足 |

## user_sentiment 取值

| 取值 | 使用说明 |
| --- | --- |
| 正向 | 主要表达满意、认可或推荐 |
| 负向 | 主要表达不满、抱怨、问题或流失意愿 |
| 中性 | 主要是描述、建议或事实陈述 |
| 混合 | 同时包含明确优点和问题 |
| 不明确 | 文本太短或情绪难以判断 |

## priority 取值

| 取值 | 判断标准 |
| --- | --- |
| P0 | 阻断核心体验，涉及严重幻觉、账号/支付/隐私风险、崩溃、卸载或退款意愿 |
| P1 | 明显影响使用体验或高频核心场景，但不一定完全阻断 |
| P2 | 轻微建议、体验优化、正向场景沉淀或低风险问题 |

## user_need 写作规则

- 写成用户真实需求，不写成产品方任务。
- 尽量使用“希望……”或“需要……”表达。
- 不写最终结论，不扩大到全部用户。

示例：

- 好：`希望获得准确可信且可验证的信息`
- 好：`希望首次打开时知道适合自己的核心功能入口`
- 不好：`用户都觉得产品不好用`

## product_suggestion 写作规则

- 必须和评论内容直接相关。
- 必须具体、可执行，使用清晰动作描述。
- 可以包含产品、运营、内容、模型评估或商业化建议。
- 写成建议动作和预期方向，效果留给后续指标验证。

示例：

- 不好：`优化用户体验`
- 好：`为新用户增加场景化入口引导，按写作、搜索、学习、办公展示首个可执行任务。`
- 不好：`提升模型质量`
- 好：`收集回答不准和幻觉样本，按事实错误、引用缺失、指令跟随失败建立模型质量复盘表。`
- 不好：`改善会员体验`
- 好：`在会员页补充免费版与会员版核心权益对比，并说明高频场景下的使用限制。`

## notes 写作规则

- 记录样本是否为示例行、标签不确定点或次要问题。
- 不写长篇分析。
- 记录评论中能够直接支持的信息。

## 样本说明

初始备选应用千问的公开评论中有效 AI 产品体验反馈较少，相关样本已归档，主分析样本聚焦豆包、DeepSeek 和 Kimi。后续新增样本时，优先选择直接指向 AI 回答质量、任务执行、交互体验、性能稳定性、会员限制、账号隐私或产品定位问题的评论。
